# EBAC-T4 — CTO Capsule

**Deterministic Trace-Bound Authorization for High-Risk EHR Writes**

EBAC-T4 demonstrates a deterministic authorization gate for high-risk EHR state-changing actions generated by agentic systems.

The reference implementation focuses on a single operation:

ehr.submit_order

The goal is not to evaluate clinical correctness, but to determine **whether an agent is authorized to execute a state-changing action at runtime**.

---

## Problem

Agentic systems can generate syntactically valid API calls capable of modifying external systems such as EHRs.

Existing control layers typically address adjacent concerns:

- **RBAC** — who is allowed in principle  
- **Clinical Decision Support (CDS)** — clinical correctness checks  
- **Policy engines (e.g., OPA)** — structured policy evaluation  
- **Schema validation** — payload structure validation  

These layers typically do not enforce a strict runtime contract tying **execution authority** to a **fresh and complete evidence cycle**.

For high-risk writes, this creates a control gap.

---

## What EBAC-T4 Implements

EBAC-T4 introduces a deterministic **pre-execution authorization gate** that validates whether a state-changing action may proceed.

Execution authority is granted **per evidence cycle**, not by role alone.

A high-risk write is allowed only if:

1. Required traces are present  
2. Traces are fresh at gateway receive time  
3. Cross-trace identity invariants hold  
4. HITL approval is bound to the exact payload + trace identifiers  

Otherwise the action is downgraded or halted.

---

## Deterministic Decision Model

The authorization gate returns one of three outcomes:

| Decision | Meaning |
|---|---|
| **ALLOW** | State change permitted |
| **DOWNGRADE** | Context incomplete → downgrade to safe workflow |
| **HALT** | Integrity violation or blocked execution |

Downgrade invalidates the execution lineage for the attempted action.

Reason codes distinguish incomplete context, integrity violations, replay attempts, and post-downgrade bypass.

---

## Minimal Evidence Contract

For `ehr.submit_order`, the required traces are:

- `read_patient`
- `read_allergies`
- `read_current_medications`

Conditions:

- traces must be fresh (TTL enforced at gateway receive time)
- patient identity must match across traces
- HITL approval must bind to the exact action context

Natural-language explanations from the agent are ignored.

---

## Reference Implementation

The repository provides a **minimal deterministic authorizer** implementing:

- trace completeness checks
- trace freshness validation
- cross-trace identity invariants
- HITL binding validation
- retry ceilings and downgrade locks
- deterministic decision mapping

The implementation intentionally focuses on a single high-risk action to demonstrate the authorization semantics defined in the specification.

## Architectural Roadmap

The broader gate model includes five authorization stages:

- Gate 0 — Tier Sanity  
- Gate 1 — Privilege Check  
- Gate 2 — Evidence Check  
- Gate 3 — Execution Graph Governance *(planned)*  
- Gate 4 — Zero-Trust Inter-Agent *(planned)*  

Version v0.2 implements **Gates 0–2**.

Autonomy Levels (A0–A4): agent autonomy graduation determining available tier ceiling — planned.

---

## Non-Goals

EBAC-T4 does **not**:

- replace RBAC
- replace clinical decision support
- validate clinical correctness
- implement a generic policy engine
- provide production-grade cryptographic guarantees

The repository demonstrates a **deterministic authorization artifact** for high-risk state changes.
