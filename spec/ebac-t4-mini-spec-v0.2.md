# EBAC-T4 Mini Specification v0.2

**Deterministic Trace-Bound Authorization for High-Risk EHR Writes**

Status: **Frozen (v0.2)**  
Scope: **Single high-risk EHR action**

---

# 1. Purpose

EBAC-T4 defines a deterministic authorization gate for high-risk EHR state-changing actions generated by agentic systems.

The specification demonstrates how execution authority for a high-risk action can be granted **per evidence cycle**, rather than by role alone.

The goal is not to determine whether a clinical decision is correct, but whether a system is authorized to execute a state-changing operation at runtime.

---

# 2. Scope

Version v0.2 covers a single operation:

ehr.submit_order

The authorization gate evaluates whether the action may proceed based on:

- required traces
- trace freshness
- cross-trace invariants
- human approval binding

The gate operates **before execution**.

---

# 3. Design Principle

Execution authority is granted **per evidence cycle**, not by role alone.

An action may proceed only if the authorization gate validates that all required evidence conditions are satisfied at evaluation time.

If the evidence cycle is incomplete or invalid, the action must not execute.

---

# 4. Authorization Outcomes

The authorization gate produces one of three deterministic outcomes.

| Decision | Meaning |
|---|---|
| **ALLOW** | State-changing action may proceed |
| **DOWNGRADE** | Evidence incomplete or insufficient |
| **HALT** | Integrity violation or execution blocked |

Reason codes distinguish specific causes such as:

- incomplete context
- integrity violations
- replay attempts
- post-downgrade bypass attempts

A downgrade invalidates the execution lineage for the attempted action.

---

# 5. Evidence Model

Authorization is based on a set of independent traces representing system state at the time of decision.

For `ehr.submit_order`, the required traces are:

- `read_patient`
- `read_allergies`
- `read_current_medications`

Each trace must include:

- trace identifier
- timestamp
- payload

Traces represent independently obtained observations of the system state.

---

# 6. Freshness Policy

Trace validity is determined by **gateway receive time**.

Agent-provided timestamps are not authoritative.

Freshness is evaluated using:

gateway_receive_time − trace_timestamp ≤ TTL

If any required trace exceeds the allowed TTL window:

Decision = DOWNGRADE

---

# 7. Cross-Trace Invariants

All traces must refer to the same patient identity.

Minimum invariant:

trace.patient_id equality across traces

If identity mismatch occurs:

Decision = HALT
Reason = INTEGRITY_VIOLATION

Additional domain invariants may be applied (e.g., allergy conflicts).

---

# 8. Human-in-the-Loop Binding (HITL)

High-risk actions require explicit human approval.

Approval must be bound to:

- the exact action payload
- trace identifiers used during authorization

The approval must correspond to the specific action context being evaluated.

Replay of approvals from previous contexts must fail validation.

---

# 9. Trace Completeness

All required traces must be present.

Missing traces result in:

Decision = DOWNGRADE
Reason = INCOMPLETE_CONTEXT

The system must not attempt to infer missing evidence.

Natural-language explanations from the agent are ignored.

---

# 10. Post-Downgrade Behavior

A downgrade invalidates the execution lineage for the attempted action.

The same action instance may not be retried with the same evidence cycle.

A new attempt requires:

- a new action identifier
- a new evidence cycle
- fresh traces

---

# 11. Retry Ceiling

Authorization attempts are bounded.

Repeated attempts using incomplete or invalid evidence must not allow progressive escalation.

Retry limits apply to the semantic action lineage rather than individual tool calls.

---

# 12. Deterministic Evaluation

The authorization gate must be implemented using deterministic logic.

Authorization decisions must not rely on probabilistic inference.

In particular:

- LLM output must not determine authorization
- natural-language explanations are ignored
- all conditions must be evaluated using structured data

---

# 13. Non-Goals

EBAC-T4 does **not** attempt to:

- replace RBAC
- replace clinical decision support
- determine clinical correctness
- provide a generic policy engine
- implement production-grade cryptographic security

The specification demonstrates a **deterministic authorization artifact for high-risk state changes**.

---

# 14. Architectural Context

The broader gate model includes five authorization stages:

- Gate 0 — Tier Sanity  
- Gate 1 — Privilege Check  
- Gate 2 — Evidence Check  
- Gate 3 — Execution Graph Governance *(planned)*  
- Gate 4 — Zero-Trust Inter-Agent *(planned)*  

Version v0.2 implements **Gates 0–2**.

Autonomy Levels (A0–A4): agent autonomy graduation determining available tier ceiling — planned.

---

# 15. Acceptance Criteria

An implementation conforms to EBAC-T4 v0.2 if it demonstrates:

- deterministic authorization decisions
- trace completeness validation
- trace freshness validation
- cross-trace identity invariants
- HITL approval binding
- downgrade invalidation of execution lineage
